eBPF 的局限性有哪些？

虽然 eBPF 很强大，但它不是万能的。在开发 Antidistractor 时，我们需要面对以下挑战：

1. 协议复杂性 (H3/QUIC)

HTTP/3 使用 UDP 且握手过程加密更早。eBPF 很难像处理 TCP 那样轻松提取 SNI。如果网站强制走 HTTP/3，屏蔽难度会增加。

2. 加密客户端握手 (ECH)

现代浏览器正逐步引入 Encrypted Client Hello (ECH)。这会把域名（SNI）也加密掉。一旦 ECH 普及，所有基于 SNI 的“透明屏蔽”都会失效。

3. 内核版本依赖

不同 Linux 发行版的内核版本不同。为了让 eBPF 程序在所有机器上运行（CO-RE），目标机器的内核需要开启 BTF 支持（通常内核 > 5.4）。

4. 分段包处理

如果一个 TLS 握手包被拆分成了多个 TCP 分段，eBPF 在内核态很难进行“流重组”。这可能导致漏掉某些屏蔽请求。

我们的应对策略？

  • 降级方案： 当 eBPF 无法识别时，配合 Nftables 的 IP 屏蔽作为兜底。
  • 开发语言： 使用 Go (Aya/Cilium) 或 Rust (Aya)。Rust 对 eBPF 的支持非常现代，有助于处理这些复杂的边界情况。

这些限制是否会动摇你选择 eBPF 的决心？还是说我们依然坚持这个硬核方向？
